git实践一

Posted on 2020-10-28 In 技术 Disqus:

Git的合并分支包括合并远程分支和合并本地分支，其实本意是一样的。只不过拉远端分支的时候多了一步git fetch，就是把所有分支的代码从云拉到本地缓存，但是不做合并。关键是合并，说到合并的时候就要说到这最纠结的两兄弟了。

git merge

git merge全称其实是git merge –ff,ff就是fast forward——快进！
在合入分支没有任何提交的情况下，它的指针顺着被合入分支的提交线一路下去指向最后一个节点上。它不会产生merge commit，所以是快进，非常轻便。(右图所示)

那这时候会问，我要知道合入的细节怎么说，那就得说git merge –no-off。(如左图所示)
它的作用就是把被合入分支的详细提交情况都显示出来，然后在最后产生一个merge的新commit，让head指向它。我们就可以清楚的看见这次的merge是怎么来的。

总结下，ff是head最终指向被合入分支的最后一次提交，而no-ff是指向他们的合入。如果此时要reset，这两种方式的reset也是不同，第一种回到了被合入分支的commit点(并不是本分支的)，而第二种才是真正回到了当前分支的上一个提交点。

以上这种情况有前提的，前提是合入分支没有任何合入。下面这种情况是我们绝大多数的merge情况。

在两个分支基于最近公共点都有新提交，这样git merge会将两个分支的最后节点做合并产生新的节点，并且生成一个merge的commit记录。它是严格按时间线记录了分支的提交记录。这样的缺点就是在提交线上会分叉，看起来很乱，而且不知道你的分支和目标分支的head在哪。

git rebase

参考了很多的博客，rebase的意思用变基是比较准确的。但是要遵守rebase黄金法则，no one shall rebase a shared branch.

rebase是在共同父提交分支上，把被合入分支提交原封拷到当前分支，再把当前分支的提交放弃，接着刚才的拷过来的提交产生新的提交。整个过程就是一个fast forward，不会产生merge commit。
它的本意就是让每个开发分支，都保持在一个基准上。这样在提交线上，就是一条直线，你可以清楚看见你领先于被合入分支多少个commit。对于开发分支很清晰，简洁。这就是它和merge的区别。
另外rebase可以压缩commit请求，在提pr时候，可以git rebase head～压缩commit。

这两个命令至今难以清楚的分清界限，但凡是存在的肯定是合理的。我认为还是取决于项目情况。参考我平时的使用情况是这样的：

  1. 从远端同步到master使用git pull –rebase，可以减少一些无用的merge提交记录，让提交线变得简单，另外我的master是只拉代码，是绝对不会提交pr的，所以不会同开发者冲突！
  2. 在feature分支合并是git rebase master，会清晰看出来你的分支和master的指针指向的。如果是共同分支开发，肯定是使用merge。
